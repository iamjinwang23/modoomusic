# MONO(모두의 노래) 시스템 아키텍처 정의서

> 문서 종류: **시스템 아키텍처 정의서** (System Architecture Definition)
> 작성일: 2026-07-02 · 버전: v1.0
> 목적: MONO의 시스템 구성·기술 스택·서버 인프라·개발 범위를 정의·공유

---

## 1. 개요

MONO는 음악 지식이 없어도 누구나 몇 초 만에 노래를 만드는 AI 음악 창작 서비스입니다.
단일 웹 서비스이지만 로그인·결제·AI 연동·소셜·커뮤니티·알림·보안·법적 준수를 모두 갖춘 완결형 시스템으로 구성되어 있습니다.

### 규모 (2026-07-02 코드베이스 실측)

| 항목 | 규모 |
|---|---|
| 전체 코드 | 약 30,000줄 / 249개 파일 |
| 화면(페이지) | 31개 (사용자 21 + 관리자 백오피스 10) |
| 백엔드 API 엔드포인트 | 81개 (관리자 전용 26 포함) |
| 데이터베이스 테이블 | 약 24개 |
| DB 마이그레이션(구조 변경 이력) | 53회 |
| UI 컴포넌트 | 84개 |

## 2. 시스템 구성 (아키텍처)

MONO는 5개 계층으로 구성되며 실시간으로 연동됩니다.

| 계층 | 구성 | 역할 |
|---|---|---|
| **프론트엔드** | Next.js(App Router)·React·TypeScript | 사용자 화면·상호작용. 반응형(모바일·PC) |
| **백엔드(API)** | Next.js Route Handlers (서버리스 함수) | 요청 처리·인증 가드·비즈니스 로직 호출 (81개 엔드포인트) |
| **데이터/인증** | Supabase (PostgreSQL·Auth·Storage) | 데이터 영속·RLS 보안·파일 저장·소셜 로그인 |
| **AI 엔진** | MiniMax (Music / Hailuo) | 음악·6초 비디오 커버 생성 (비동기 작업 → 폴링·웹훅) |
| **인프라** | Vercel | 자동 배포·서버리스 스케일링·CDN |

### 요청 흐름 예시 (곡 생성)
`사용자(화면) → API(인증·크레딧 확인) → MiniMax 호출 → 백그라운드 생성 → 완료 시 Supabase 저장 + 알림(인앱·웹푸시) → 화면 갱신`

- 서버·프론트가 동일 저장소에서 **TypeScript 타입을 공유**하여 계약(스키마) 불일치를 컴파일 단계에서 차단합니다.
- 곡 생성 등 장시간 작업은 **백그라운드 비동기 처리**로, 사용자가 페이지를 떠나도 완료됩니다.

## 3. 기술 스택

| 영역 | 기술 | 비고 |
|---|---|---|
| 프레임워크 | Next.js 16 · React 19 | App Router 기반 SSR/CSR 혼합 |
| 언어 | TypeScript 5 | 전 계층 정적 타입 |
| 스타일 | Tailwind CSS 4 | 디자인 시스템·일관성 |
| DB·인증·스토리지 | Supabase (PostgreSQL) | RLS 기반 접근 제어 |
| 소셜 로그인 | 구글 · 애플 · 네이버 · 카카오 (4종) | OAuth |
| AI 생성 | MiniMax (Music-2.6 / Hailuo 영상) | 음악·비디오 커버 |
| 결제 | PortOne(포트원) | 크레딧 충전 (게이팅 상태) |
| 알림 | 웹 푸시(VAPID) · PWA | 설치형 앱 + 푸시 |
| 분석·운영 | GA4 · Vercel Cron(3종) | 행동 분석·자동 정리 작업 |

전 계층을 업계 표준·검증 기술로 구성하여 안정성과 유지보수성을 우선했습니다.

## 4. 개발 범위 (구현 완료 기준)

각 기능은 **기획 → 개발 → 데이터 설계 → 보안 → 정책·약관 → 도움말**까지 완결(vertical slice)하여 축적했습니다.

**① 코어 — 음악 창작**
- AI 작곡(가사 직접 입력 / AI 자동 가사 / 심플·고급 모드), 인스트루멘탈
- 곡 커버 이미지, 6초 비디오 커버
- 백그라운드 생성 + 완료 알림

**② 소셜 — 발견·공유**
- 둘러보기·검색·장르 탐색·추천 크리에이터
- 좋아요·댓글(대댓글)·팔로우·컬렉션
- 공개 게시 / 비공개 링크 공유

**③ 커뮤니티 — 카페형** *(구현 완료, 정식 오픈 게이팅 중)*
- 다중 커뮤니티 개설·가입·매니저 운영
- 글(이미지·투표·곡·링크 자동 임베드)·댓글·좋아요·신고
- 커뮤니티 활동 알림, 인기글·순위·스토리 등 다양한 노출

**④ 비즈니스·운영**
- 크레딧(일일 충전 + 초대 보너스 + 유료 충전)
- 결제·환불·영수증 / 관리자 백오피스
- 회원 탈퇴(7일 복원)·신고 처리·금칙어 필터

**⑤ 기반**
- 4종 소셜 로그인, PWA·웹 푸시
- SEO·검색엔진 등록(구글·네이버)
- 이용약관·개인정보처리방침·운영정책
- 자동 정리 작업(크론) 3종

**⑥ 관리자 백오피스** *(운영 전용, is_admin 권한)*
- **대시보드**: 핵심 지표 집계
- **회원 관리**: 검색·목록·상세, 정지/해제, 관리자 권한 부여/회수, 강제 탈퇴
- **결제 관리**: 결제 내역·취소·환불·PG 동기화
- **콘텐츠 관리**: 곡 게시 취소·삭제
- **신고 처리**: 곡·댓글·커뮤니티 글 신고 큐 검토·조치
- **금칙어 관리**: 등록·수정 (전 UGC 필터에 반영)
- **공지(What's New)**: 마크다운 에디터·예약 발행·전체 알림 발송
- **크레딧 지급**·**AI 모델 설정**
- **감사 로그(Audit)**: 관리자 행위 기록·CSV 내보내기
- 규모: 관리자 화면 10개 · 전용 API 26개

## 5. 비기능 요구사항 (보안·안정성·확장성)

- **보안**: Supabase RLS(행 수준 접근 제어)로 타 사용자 데이터 접근 차단, 결제 위·변조 검증, 금칙어·신고 기반 모더레이션
- **데이터 정합성**: 좋아요·댓글·멤버 카운트를 DB 트리거로 원자적 관리, 좋아요 알림 중복 방지
- **안정성**: 곡 생성 실패 시 크레딧 자동 환불, 만료 푸시 구독 자동 정리
- **확장성**: 서버리스 구조로 트래픽에 따라 자동 스케일링
- **법적 준수**: 개인정보 수집 고지, 전자상거래법 사업자 정보, 환불 정책 명문화

### 처리 복잡도 예시 — "커뮤니티 글 좋아요"
1. 로그인·멤버 자격 확인 → 2. 좋아요 저장·중복 방지 → 3. 좋아요 수 갱신(트리거) → 4. 글쓴이 알림(인앱+푸시) → 5. 본인·중복 알림 차단 → 6. 화면 즉시 반영
단일 인터랙션이 6단계 서버 처리로 구성됩니다.
